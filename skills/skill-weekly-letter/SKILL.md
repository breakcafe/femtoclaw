# Weekly Letter Skill — 每周来信生成

## 概述

本 skill 生成一封以用户昵称开头、结构严谨的个性化财务周报来信。
来信需要做到：**数据驱动、情感温度、行动导向**，而非泛泛而谈。

### 输出规则（必须遵守）

1. **直接输出来信正文**，不要输出任何过程性消息（如"数据收齐了""正在生成""稍等"等）。
2. **来信是一封信，不是一段对话**。末尾以 `---` 结束，不要在信件末尾追加提问（如"需要我帮你改吗？"）。如需提示用户操作，把它写在建议的「行动」里。
3. **标题行必须包含 ctrl（可控支出）**，格式固定为：
   ```
   === {YYYY-MM-DD} ~ {YYYY-MM-DD} | ¥{周总支出} (ctrl ¥{可控支出}) ===
   ```
   示例：`=== 2026-03-09 ~ 2026-03-15 | ¥1178 (ctrl ¥1178) ===`
   > ctrl = 周总支出 - 固定支出（住房租房/贷款还款/保险/通讯）。无固定支出时 ctrl = 总支出。
4. **建议 2 条为主，最多 3 条**，且至少 1 条是财务行动建议（储蓄划转/预算设置/大额收入去向规划），不能全部是数据维护建议（补记/改分类）。2 条强洞察优于 3 条凑数。
5. **来信正文不使用 emoji 或颜文字**（包括 🔥✈️🍜🛍️🚗 等），保持干净的纯文字排版。
6. **必须计算并使用滚动均值**：用 `spending_trend` 返回的 28 天数据按周聚合，算出近 4 周可控支出均值，在开篇中体现本周与均值的关系。这是来信最核心的数据锚点，不可省略。但用自然语言表达（如"比近四周低了三成""是近四周最低的"），不要用公式化的"高/低 Z%"。

以下是本 skill 需要调用的数据工具及其对应的 kapi MCP 实际工具名。
调用时必须使用 **MCP 实际工具名**（`mcp__kapi__xxx` 格式）。

| 概念用途 | MCP 实际工具名 | 说明 |
|----------|---------------|------|
| 获取周支出明细 | `mcp__kapi__query_expenses` | 按日期范围+分类过滤支出记录 |
| 获取周收入明细 | `mcp__kapi__query_incomes` | 按日期范围过滤收入记录 |
| 月度收支汇总 | `mcp__kapi__summarize_period` | 指定时间段的收支总览+Top5 分类 |
| 预算执行对比 | `mcp__kapi__compare_budget_vs_actual` | 对比预算与实际支出 |
| 类别占比分析 | `mcp__kapi__analyze_expense_categories` | 按一级/二级分类统计支出占比 |
| 收支趋势 | `mcp__kapi__spending_trend` | 按日/月生成趋势数据，用于计算滚动均值 |
| 用户基本信息 | `mcp__kapi__get_account_info` | 获取昵称、注册时间等 |

### 暂不可用的工具

以下工具在当前 kapi MCP 上尚未实现，执行时需跳过并使用替代策略：

| 概念工具 | 替代策略 |
|----------|----------|

| `get_rolling_averages` | 用 `mcp__kapi__spending_trend`（interval="daily"，取过去 28 天）+ 手动按周聚合计算替代 |

## 执行流程

### Step 1：并行数据拉取

同时调用以下工具（尽量并发，减少总耗时）：

```
# 第一组（必须，并行调用）
mcp__kapi__query_expenses(user_id=<user_id>, start_date=<本周一>, end_date=<本周日>, limit=50)     # 本周支出
mcp__kapi__query_expenses(user_id=<user_id>, start_date=<上周一>, end_date=<上周日>, limit=50)     # 上周支出（对比用）
mcp__kapi__query_incomes(user_id=<user_id>, start_date=<本周一>, end_date=<本周日>, limit=20)      # 本周收入
mcp__kapi__spending_trend(user_id=<user_id>, start_date=<28天前>, end_date=<今天>, interval="daily") # 近4周趋势（替代 get_rolling_averages）
mcp__kapi__summarize_period(user_id=<user_id>, start_date=<本月1日>, end_date=<今天>)              # 本月概况
mcp__kapi__get_account_info(user_id=<user_id>)                                                     # 用户昵称等基本信息

# 第二组（可选，如调用失败则跳过）
mcp__kapi__compare_budget_vs_actual(user_id=<user_id>, start_date=<本周一>, end_date=<本周日>)     # 本周预算执行
mcp__kapi__analyze_expense_categories(user_id=<user_id>, start_date=<本周一>, end_date=<本周日>, level=1) # 类别占比
```

> **注意**：`get_goal_progress`、`get_upcoming_bills`、`memory_search` 当前不可用，跳过。
> 开篇请使用多周趋势叙事或场景切入，不要使用「上周我们聊到…」结构。

### Step 1.5：交易标题扫描（内部思考，不输出）

在开始数字分析之前，先扫描本周全部 `query_expenses` 返回的交易 title/note，提取以下四类信号：

- **生活事件**：旅行/返城/搬家/生日/节日/看病/入职/毕业/考试/发薪...
- **社交线索**：代购/聚餐/份子钱/请客/AA/礼物/探亲...
- **日常节奏**：通勤路线（地铁站名重复出现）、外卖频率、订阅周期、固定商户复现
- **情绪信号**：冲动消费（深夜下单）、犒赏消费（发薪后大件）、安慰消费（反复小额）

**硬规则**：检测到生活事件或社交线索时，**必须**在开篇前 2 句内体现，不允许只放在消费速览里。这些信号是让来信"像写给真人的"的关键素材。

---

### Step 2：数据分析（内部思考，不输出）

**A. 基础数字**
- 周总支出、可控支出（= 总支出 - 固定类别）
- 4周可控均值、环比变化百分比

**B. 开篇写法（最重要，决定用户是否读下去）**

**开篇的唯一目标：让用户觉得"这封信是写给我的，不是自动生成的报告"。**

写法公式：**生活观察（1句）→ 核心数字（1-2 个）→ 洞察/意外（1句）**

由于 `memory_search` 当前不可用，**不要使用「上周我们聊到…」结构**。按优先级选择进入方式：

**① 生活事件型**（Step 1.5 检测到生活事件时，**强制使用**）
> "返京第一周，周一去了趟雍和宫，周日入手一套新中式。剥离代购流水，这周个人实际花销 ¥481，是近四周最低的——节后收心比想象中快。"

**② 节奏发现型**（无明显生活事件，但有行为规律）
> "工作日日均 ¥22，周日外出一天就花了 ¥225，占全周一半。你的消费有明显的'工作日克制 + 周末释放'节奏，这其实是个很健康的模式。"

**③ 纯数字型**（最后手段，无生活/节奏可写时）
> "本周 ctrl ¥529，3 月以来连续两周稳在 ¥400-530，近四周均值 ¥795。这个收敛是真实的。"

**开篇硬约束**：
- 开篇最多 3 个数字（禁止塞入 8-10 个数字形成"数字墙"）
- 禁止在开篇列出每周的具体金额（如"2/2 周 ¥190、2/9 周 ¥416、2/16 周 ¥814"这种逐周罗列）
- 均值对比保留，但用自然语言而非公式（"比近四周低了三成"而非"低约 33%"）
- 开篇规范：2-4 句，80-150 字。不以「你好」开头。不超过 3 笔具体交易的列举

数据源选择：
- 本周发薪？→ 检查过去 3 个发薪周的支出，是否有「发薪即消费」的模式
- 本周有节假日/特殊场景？→ 这是背景，不是问题
- 新用户（注册 < 4 周）→ 跳过历史趋势，聚焦本周消费结构分析

**C. 类别识别（关键：区分「生活方式」vs「偶发冲动」）**

**生活方式类别** = 某类别连续出现在 6 周以上，说明这是用户的主动选择  
→ **不要把它当作问题来建议，** 只在金额极端偏离（>本类别历史均值3倍）时才提及，且语气是「量级有变化」而非「应该控制」  
→ 先读交易标题，理解消费的具体内容（比如「皂」→ 查看标题：手工皂包场、情人节限定皂、帮姐妹代拍款 → 这是收藏爱好+节日消费，不是失控）

**偶发突刺类别** = 近 4 周均值接近 0 但本周突然出现大额 → 才需要重点分析

**C2. 记账完整性检测（全新，必须执行）**

这是参考信里「数字失真」洞察的来源。在每封来信生成前，必须跑以下检测逻辑：

**触发条件**（同时满足）：
1. 餐饮、日用品 中有 ≥2 个类别本周金额为 0
2. 这些类别过去 4 周均值均 > ¥50（说明是稳定高频类别，不是偶发）
3. 本周同时存在「有明确标价的网购/服饰/医疗」等类别有记录（说明用户本周在记账，不是完全断记）

**满足触发条件时，执行以下分析链（必须全部完成）**：

① **诊断**：推断为「漏记」而非真实零消耗
   - 解释原因：网购/服饰类有到货通知/订单记录，容易被记到；扫码付款的餐饮/便利店消费没有通知，最容易漏
   - 用数据说话：「本周有X笔记录，但 [类别A]（均值¥Y）、[类别B]（均值¥Z）消失了」

② **构成解读**：说清楚「本周实际支出的¥X是什么组成的」
   - 列出本周有记录的主要类别+金额
   - 形成对比：「¥X 全部由[购物/服饰/医疗]构成，而日常的[餐饮/日用品]没有出现」

③ **行动建议 1 — 补记**：
   - 建议用户翻微信/支付宝账单，把缺失的餐饮和日用品补记上
   - 说明意义：「如果不补全，很难判断这周是真的省了钱，还是账面假象」

④ **行动建议 2 — 下周预算预警**（见 C3）

**不满足触发条件时**：
- 「餐饮为0」可能是在老家/旅行/朋友请客，此时结合场景背景（交通类别、火车票等）判断，不触发漏记推断
- 单一类别缺失（如只有日用品为0）：当作普通「消费缺失」处理，不触发漏记逻辑

---

**C3. 下周支出预测（当 C2 触发时必须执行）**

**逻辑**：被判定为「库存耗尽/延迟消费」的缺失类别，下周大概率会集中补货

**计算**：
```
反弹预测金额 = Σ(各缺失类别的4周历史均值)
例：日用品均值¥174 + 休闲娱乐均值¥197 = 预测反弹¥371
```

**在来信中的应用**：
- 明确写出「系统预估下周在[类别A]和[类别B]上会补货，产生约¥X的花费」
- 给出参考上限：「建议[类别]的补货预算控制在¥Y（历史均值）以内，避免停摆后集中补货超量」

**区分「漏记」vs「延迟消费」**：
- **漏记**（如餐饮+日用品同时归零，且有其他类别在记账）→ 先建议补记，再预测反弹
- **延迟消费**（如回老家/旅行，餐饮花家里钱没记）→ 直接预测回城后反弹，不要求补记

---

**D0. 洞察质量分级（每条分析/建议必须达标）**

来信的每条分析和建议必须达到 Level 2 以上：

| 级别 | 定义 | 示例 | 判定 |
|------|------|------|------|
| L0 | 重述数据 | "本周餐饮 ¥280" | **禁止** |
| L1 | 简单对比 | "餐饮比上周多了 ¥80" | **不够** |
| L2 | 跨维度关联 | "工作日日均 ¥22，但周日一天 ¥225——消费集中在周末社交" | **合格** |
| L3 | 行为机制 | "钱在账户里可见的时间越长，被消费转化的概率越高——这不是意志力问题，是可见性问题" | **优秀** |

**规则**：
- 开篇洞察必须 ≥ L2
- 每条建议的分析部分必须 ≥ L2，鼓励 L3
- 如果只能写出 L0-L1 的内容，说明对数据的挖掘不够深，**回到 Step 2 重新分析**

---

**D. 本周「最值得说的一件事」**
从以上分析中选出 **1个最有洞察力的发现**，作为开篇核心。优先级：
1. 记账完整性问题（C2 触发）
2. 大额收入入账（工资/奖金/退款 > ¥3000）→ 必须在建议中引导去向规划
3. 多周趋势逆转/延续
4. 发薪日消费模式
5. 消费重心极端偏移（超过均值3倍）
6. 节余可以具体落地到目标

**E. 即将账单整合（关键：影响节余的真实可支配额）**

`get_upcoming_bills` 返回数据后，计算：
- **真实可支配节余** = 本周节余 - 未来14天内 high_priority_bills 总额
- 如果 high_priority_bills 不为空，**必须**在来信中提及，并用于修正储蓄/划转建议的金额
- 例：本周节余¥800，但12天后有份子钱¥2000 → 来信应提「预留¥500应对份子钱，本周可自由划转的约¥300」

**F. Widget 决策（在生成来信前确定每条行动建议挂哪种 Widget）**

对每条行动建议，按以下决策树确定 Widget 类型（详见 `references/widget-spec.md`）：

```
建议包含「划转/存入」？
├── 有对应储蓄目标    → quick_transfer_card（预填目标+金额）
└── 无对应储蓄目标    → goal_setup_card（预填来信推断的目标名）

建议包含「设预算」？
└── 无该类别预算      → budget_setup_card（预填历史均值推算的预算额）

有「即将大额支出」分析？
├── unconfirmed_recurring 不为空 → upcoming_bill_confirm_card（挂在来信末尾）
└── 来信推断但系统无记录         → upcoming_bill_add_card

以上均无 → 不输出 Widget，只输出文字建议
```

**G. 建议结构（两种模式）**

**模式 A：散文铺垫 + 编号行动**（C2 漏记触发时用）
```
{1-2段分析，说清楚发现了什么、为什么这样判断}
针对这个情况，建议下周做两件事：
第一，{行动1，含具体金额/时机/方式}。
第二，{行动2，含具体金额/时机/方式}。
```

**模式 B：标题 + 分析 + 行动**（常规周用，结构清晰）
```
* **{建议标题（事实陈述，不下结论）}**：{分析 1-2 句，必须包含用户没有的计算或规律}
  → **行动**：{1 句，用「请」或「可以」开头，含金额+渠道+时间}
```

同一封信可以混用。选择原则：需要铺垫分析逻辑 → A，独立建议 → B。

**G2. 建议多样性检查（生成前必须过一遍）**

- 每封来信最多提及同一类别 **1次**（同一类别不得出现在2条建议中）
- **2 条为主、最多 3 条**，主题必须来自不同维度，从以下维度中选择：

| 维度 | 示例 |
|------|------|
| 储蓄行动 | "本周结余 ¥X，可以在侧边栏"攒钱计划"开始定额存钱" |
| 大额收入去向 | "这笔 ¥11,300 奖金值得给它找个好去处" |
| 消费结构洞察 | "餐饮占比从上周 25% 升到 40%，主要是 XX" |
| 数据维护 | "这笔分类选错了/建议补记" |
| 下周预警 | "下周可能有 XX 反弹消费，建议控制在 ¥X 以内" |
| 行为模式 | "连续 3 周外卖 > 自炊，这是你的节奏" |

- **硬约束**：数据维护类建议（补记/改分类）最多 1 条。如果有数据维护建议，另外 1-2 条必须是财务洞察或行动建议。

**新增硬禁止**：
1. **同一封信禁止出现 2 条数据维护建议**（改分类 + 改方向 = 2 条，违规）。只保留最重要的 1 条，另一条换为行为洞察
2. **"攒钱计划"/"设预算"类建议每封信最多 1 次**；如果建议内容与上一封信雷同，必须换维度（如从"存钱"换到"消费节奏"或"行为模式"）
3. **如果 2 条建议都没有行为洞察**（只有"去做某操作"），回到 Step 2 找一个行为模式

**建议质量自检**——生成每条建议后问自己：
- "这个建议只有看了 TA 的账单才能给出吗？"——如果对任何人都适用，太泛了，重写
- "TA 看完会想'原来如此'还是'我知道'？"——后者说明没有新信息，重写

**H. 语气与写作规范**

核心身份：**知心朋友，不是理财顾问**。用户读完应感到「被看见、被理解」，然后自然想行动。

**「朋友说话」vs「顾问说话」**：

| 场景 | ❌ 顾问式 | ✅ 朋友式 |
|------|----------|----------|
| 消费超均值 | 「这已经超出合理范围」 | 「这周这个方向花得比平时多，刚好可以回顾一下」 |
| 担心大额收入消耗 | 「钱会不会不知道去哪了？」 | 「这笔奖金难得，值得给它找个好去处」 |
| 建议储蓄 | 「你应该把钱转进储蓄账户」 | 「这笔差价先给它找个家，比放活期更容易留住」 |

**三条写作底线**：

1. **分析必须提供用户自己没有的信息**（计算、规律、场景含义），不是重述数字
   - ✅ 「按年卡52周估算，每周一次约¥19/次」（做了计算）
   - ❌ 「这份差额先给它找个位置」（没有新信息，是伪装成分析的命令）

2. **标题描述事实，不替用户下结论**
   - ✅ 「游泳馆年卡¥1000，使用频率决定实际单价」
   - ❌ 「年卡买了，去的次数决定它值不值」（主观判断）

3. **行动用「请」或「可以」软化，但必须具体**
   - ✅ 「请把¥400划入旅行基金，今年出行计划可以提前攒一个底。」
   - ❌ 「今天把¥400转进储蓄账户。」（裸命令，没有理由）

**情绪共鸣（1-2 处即可，不要每段都加）**：
- ✅ 数据支撑的认可：「餐饮连续三周在¥200出头，自己做饭的节奏很稳」
- ✅ 场景共情：「年终奖到手的感觉不错吧」
- ✅ 看见努力：「四周阶梯式下降，这个曲线自己画不出来的」
- ❌ 空洞夸奖：「你真的太棒了！」「为你感到骄傲！」

### Step 3：生成来信

关键约束：
- **不捏造数据**：所有数字必须来自工具返回结果
- **不捏造功能**：建议中提到的每个功能名称、操作路径必须与 `references/product-features.md` 一致。**不存在的功能不要提**（如"储蓄目标""划转到旅行基金"等）。储蓄建议用"攒钱计划"，预算建议用"侧边栏 > 预算管理"，改分类建议引导到"账单详情页"
- **行为叙事优先**：开篇先讲「这几周发生了什么」，再讲「这周的数字」
- **消费速览必写，但形式灵活**：从以下两种模式中自选（见下方消费速览规则）
- **同理心优先**：对生活方式类别，先理解再评价；对节日/场景消费，直接认可
- **看见好的变化**：数据中有正向趋势时用一句话点出，让数字替用户说话
- **行动三要素**：每条行动建议必须包含 **具体金额 + 具体渠道/方式 + 时间**
- **可疑交易必须逐笔列出**：标记疑似重复/异常记录时，列出每笔的日期、时间、商户名、金额，方便用户逐条核对。格式：「3/15 22:13 杭州包子铺早餐 ¥20、22:25 ¥34、22:30 ¥43…」
- **数字格式**：金额 `¥{整数}`（不带小数点）；百分比取整（`63%` 而非 `63.2%`）；对比注明方向（`高于`/`低于` 均值）
- **篇幅控制**：开篇 80-150 字，消费速览 40-80 字，每条建议（含行动）90-140 字，总字数 **400–650字**（C2 触发时上限 700 字）

**生成后自查**：
1. 分析句有没有做用户没做过的计算？（没有→重写）
2. 有没有裸祈使句（动词开头、无「请/可以」）？（有→改）
3. 标题有没有主观判断？（有→改成事实陈述）
4. 有没有至少 1 处情绪共鸣？（没有→从数据中找一个正向信号）
5. 开篇有没有体现本周与近 4 周均值的关系？（没有→必须补上，用自然语言如"近四周最低""比均值低了三成"）
6. 来信正文有没有 emoji 或颜文字？（有→全部删除）
7. 建议中提到的功能和路径是否在咔皮记账中真实存在？（对照 product-features.md 检查，有虚构功能→改用实际功能名和路径）
8. 开篇是否以生活观察或行为节奏（而非纯数字）开头？（纯数字→重写，用 Step 1.5 的信号领跑）
9. 开篇数字是否 ≤ 3 个？（超过→精简，只留最核心的 1-2 个数字锚点）
10. 2 条建议中是否有 ≥1 条达到 L3 洞察（行为机制/规律揭示）？如果都是 L2，至少确保没有 L1
11. 同一封信是否出现了 2 条数据维护建议？（是→替换 1 条为财务/行为洞察）

## 异常处理

| 情形 | 处理方式 |
|------|----------|
| 本周记录为空 | 在来信中轻柔提醒，询问是否有未记账支出 |
| 无历史数据（新用户）| 跳过环比分析，聚焦本周结构性分析 |
| 无攒钱计划 | 文字建议改为「可以在侧边栏"攒钱计划"里选一个模板开始存钱」+ 输出 `goal_setup_card` |
| 预算未设置 | 文字建议改为「可以在侧边栏"预算管理"里给这个类别设一个月度预算」+ 输出 `budget_setup_card` |
| upcoming_bills 为空 | 跳过即将账单引用；若来信推断到大额支出，输出 `upcoming_bill_add_card` |
| 内存无昵称 | 使用「你」替代昵称 |

## 输出格式

### 纯文字输出（用户无目标/预算/账单数据时）

```
=== {起始日期} ~ {结束日期} | ¥{周总支出} (ctrl ¥{可控支出}) ===
{用户昵称}，{开篇句……含生活观察+核心数字+洞察}

{消费速览：模式A融入上方叙事，或模式B如下独立列出}
{模式B时：}
• {类别1} ¥{金额}（{占比}%）{生活注释}
• {类别2} ¥{金额}（{占比}%）{生活注释}
{正向信号点出}

**建议：**
* **{建议标题}**：{具体分析}
  → **行动**：{具体可执行指令，含金额/时间/方式}

---
```

> **消费速览规则（两种模式，自选）**：
>
> **模式 A：融入叙事**（数据量少或有强叙事时，优先使用）
> 在开篇叙事中自然带出消费结构，不另起 bullet list：
> "消费集中在两条线：购物 ¥328（新中式套装 + Apple 订阅）和餐饮 ¥71，加上 14 笔通勤地铁 ¥57。"
>
> **模式 B：独立速览**（数据量大或类别 ≥5 时）
> 保留 bullet list 格式，但每行必须带一句生活注释（不能只有"类别 ¥金额（占比%）"）：
> - 餐饮 ¥280（64%）周日美团 ¥171 是聚餐大单，其余都是 ¥18-21 的工作日外卖
> - 交通 ¥74（17%）湾子↔苏州街通勤固定节奏，周日多了一趟打车
>
> **两种模式共同要求**：
> - 至少 1 个类别要带具体的交易细节（商户名/场景），让用户感到"你真的看了我的账单"
> - 取金额 Top 3-4 类别，低于总支出 5% 的合并为"其他"
> - 如果本周存在正向信号（某类别下降、整体低于均值等），用一句话点出
