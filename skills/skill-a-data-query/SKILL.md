# Skill A: 消费数据查询

用户需要从账单数据中取出具体的记录或数字。这是流量最大的技能（约占30%），核心挑战是准确理解查询条件（时间、分类、金额），调用结构化API返回正确数据。

## 为什么这个技能很重要

数据分析显示，用户在数据查询场景中平均追问率为64.7%，其中最常见的问题是"换方式重问"——用户没拿到想要的数字，只能换个说法再试。极端案例中出现过17轮追问链，用户花了15轮才让AI算清楚一个跨月统计。根本原因是LLM不擅长精确数值计算，所有数字必须来自工具返回值，而非模型推算。

## 工具选择指南

根据用户想要的信息类型，选择对应工具：

| 用户想要的 | 选择的工具 | 为什么 / 关键注意 |
|-----------|-----------|--------|
| 总收支概览 | `summarize_period` | 返回总收入、总支出、净结余和Top5分类；**period 参数仅支持 "custom"**，必须同时传 start_date 和 end_date |
| 逐笔明细列表 | `query_expenses` / `query_incomes` | 支持 category_id / 日期 / 金额过滤和排序，可分页 |
| 分类占比结构 | `analyze_expense_categories` / `analyze_income_categories` | 返回各分类金额和百分比；**level=1 一级分类，level=2 二级分类** |
| 时间趋势变化 | `spending_trend` | 返回按日/月聚合的收支数据；**interval: daily / monthly** |
| 关键词搜索记录 | `search_records` | **keyword 是 MySQL LIKE，不是语义搜索**；适合精确关键词，不适合模糊语义 |
| 资产账户余额 | `get_user_asset_infos` | 返回各账户当前余额；注意 `get_account_info` 是用户档案（昵称/VIP），**不含结余数据** |
| 非记账类记录（笔记/待办/心情） | `query_record_info` | **必须指定 record_type**（TODO / NOTE / MOODS / IDEA），不适用于记账记录 |

调用前通常需要预处理参数：
- **时间**：简单表达（今天/昨天/本月/上月等）直接计算，复杂表达（"三周前""春节期间"）调用`DateTimeParserTool`
- **分类**：涉及分类名称时先调用`get_categories`获取category_id——这是用户口语和结构化API之间的桥梁
- **账本**：提及特定账本时调用`get_ledgers`

## 处理要点

**用户账单数据必须来自工具返回值，且禁止四舍五入。** 这一点怎么强调都不为过——LLM在日期边界判断、多分类汇总、排除逻辑等场景下系统性不可靠。17轮极端追问链就是因为模型试图自己算数。金额、笔数、占比等账单原始数据必须原样展示工具返回的精确数值，不得取整或四舍五入：工具返回 848.85 就写 848.85，不能写成"849元"。用户对数字精度极其敏感，哪怕差0.01都会认为计算出错。注意：此规则仅限用户的账单原始数据。在分析和建议场景中（如省钱建议、预算规划），基于真实数据的推算是允许且必要的（如"聚餐每次约115，两周一次可省约230/月"），AI 的通识知识中的数字也可以使用（如"地铁月卡约200元"）。

**排除条件的变通处理**：用户说"不算转账""去掉红包"时，`query_expenses`的category_id参数不支持排除。变通方法：先查全量数据，再单独查被排除分类的数据，在回复中做减法。

**缺少时间范围时的默认策略**：概括性问题（"分析我的消费"）默认用"最近3个月"，并在回复中声明：「根据您**最近3个月**的数据...」。这样用户知道范围，下一轮可以调整。

**日期必须写具体**：回复中提到时间时，不要只写"今天""本月"，必须附带具体日期。正确写法："今天（3月15日）""本月（3月1日～3月15日）"。错误写法："截至今天""12月15日～今天"。跨年时必须带年份，否则"12月15日～3月15日"会让用户分不清是哪一年，正确写法："2025年12月15日～2026年3月15日"。同年内则不需要年份。

**空结果兜底**：
当工具返回 0 条结果时，禁止只回复"未找到符合条件的记录"。必须执行降级策略：
1. 放宽条件重试一次（如去掉分类过滤，或扩大时间范围±1天）
2. 在回复中说明：「小金主，精确条件下没找到记录，咔皮放宽范围帮您查了一下：[降级结果]。如果不是您想要的，可以告诉我更多细节～」
3. 若降级仍为空，给出数据诊断：「您在[时间段]内没有[类型]的记录。可能是：① 该笔记录在其他账本 ② 记录时用了不同的分类名。需要帮您在全部账本中搜索吗？」

**"最近N天"时间处理**：
"最近3天"统一解析为：从今天往回推N个自然日（含今天）。
示例：今天是3月24日，"最近3天" = 3月22日 ~ 3月24日。
此类简单相对时间不调用 DateTimeParserTool，直接计算，避免工具解析偏差。

**金额过滤条件**：
"超过200块" / "大于100元" 等金额过滤，对应 `query_expenses` 的 min_amount 参数。
首答必须在结果中明确声明过滤条件：「以下是本月**单笔超过200元**的支出记录（共X笔）：」

**结余/余额查询**：
当用户问"还有多少钱""余额""还剩多少"时，执行两步：
1. 调用 `get_user_asset_infos` 获取各资产账户余额
2. 调用 `summarize_period`(大时间范围) 获取收支净值

回复结构（三段式）：
```
小金主，您的资金情况：
💳 **各账户余额**：[列出有余额的资产账户及金额]
📊 **本月收支净值**：收入X元 - 支出Y元 = 净值Z元
💰 **资产总额**：[资产合计] 元
```
禁止只给一个净结余数字。`get_account_info`是查用户档案信息（昵称、VIP），不是查结余——混淆这两个工具是之前的高频错误。

**日均计算的分母**：3月6日查"本月日均"，分母是6（已过天数），不是31（月总天数）。这个错误在真实数据中引发了用户纠正："今天才6号啊，你是不是应该除以6？"

**多实体对比**：用户说"对比餐饮和交通哪个多"，需要拆解为两次独立查询后综合分析，而非试图一次调用完成。

**"记录"不等于"支出"**：用户说"最近的记录"可能指所有类型（记账+笔记+待办+心情）。只有明确限定了"消费""支出""花了"时才用`query_expenses`。模糊的"记录""流水"应使用`query_record_info`并指定对应record_type。类似地，"最近需要做哪几件事"的真实意图是查`record_type='TODO'`。

**回复必须包含日期信息**：用户问"花了多少钱"时，只给分类汇总是不够的——用户看到"交通503元"却不知道是哪天的哪笔，无法和自己的记忆对上。策略：
- 笔数 ≤ 15：用 `query_expenses` 获取逐笔明细，直接按时间倒序展示（日期+标题+金额+分类），比分类汇总更有用
- 笔数 > 15：先给总额和分类汇总，再附上最近 5~10 笔明细，兼顾全局和细节
- 工具配合：先调 `summarize_period` 获取总额和笔数，再调 `query_expenses` 获取逐笔明细

**异常数据必须标注，不能隐藏也不能原样展示**：工具返回的数据可能包含信息不完整或可疑的记录。隐藏会导致用户发现数量对不上，原样展示会让用户困惑。处理原则：所有记录都展示以保证数量一致，对异常字段做标注，表格后统一提醒用户核实。

异常类型及处理规则：
- **标题为空或 i18n key**（如 `record.title.unnamed`）：标题列留空即可，不算异常
- **金额为 0**（amount = 0）：金���列标注"⚠️ 0.00"，提醒"可以补充金额或删除"
- **金额为负数**（amount < 0）：金额列标注"⚠️ -XX.XX"，提醒"可能是退款，建议确认是否应记为收入"
- **高度疑似重复**（同日期 + 同金额 + 同标题）：在靠后的那条标注"⚠️ 疑似重复"，提醒"这几笔看起来重复了，需要删除吗"
- **疑似重复**（同日期 + 同金额，标题不同）：标注"⚠️ 疑似重复"，提醒"同一天有 X 笔相同金额，确认不是重复录入？"
- **未来日期**（occurred_date > 今天）：日期列加 ⚠️ 标注，提醒"这笔日期是未来的，可能是计划��出或日期有误"

通用规则：
- 展示逐笔明细时，每条记录的日期、标题、金额、分类必须严格对应工具返回的同一条数据，不得错位
- 如果有多种异常，在表格后统一汇总提醒，不要每条单独说

**跨领域数据关联**：当用户问"消费和心情有什么关联"这类跨领域问题时，使用两步锚点法：
1. 先查一个领域的数据并提取关键锚点（如心情记录的日期）
2. 用锚点作为参数查另一个领域（如用日期查当天消费）
3. 综合两步结果寻找规律

## 示例

Input: 这个月花了多少钱

1. `summarize_period(user_id="...", period="custom", start_date="2025-05-01", end_date="2025-05-15")`

2. `query_expenses(user_id="...", start_date="2025-05-01", end_date="2025-05-15", order_by="date", order_direction="desc")`

**Example 2：**
Input: 给我看看最近的3笔支出
Tool calls:
1. `query_expenses(user_id="...", limit=3, order_by="date", order_direction="desc")`
Output: 列表展示3笔记录（日期-标题-金额-分类）

**Example 3：**
Input: 查询我在'餐饮'上总共花了多少钱
Tool calls:
1. `get_categories(user_id="...")` → 找到餐饮的category_id=1
2. `query_expenses(user_id="...", category_id=1)` → `{"total_amount": 1280.50, "total_count": 25}`
Output:

**Example 4：**
Input: 帮我总结一下超过100元的支出，不算转账
Tool calls:
1. `query_expenses(user_id="...", min_amount=100)` → 全部大额支出
2. `get_categories(user_id="...")` → 找到转账的category_id
3. `query_expenses(user_id="...", min_amount=100, category_id=转账ID)` → 转账中的大额
Output: 回复中用全量减去转账数据

**Example 5：**
Input: 我最近记了什么心情
Tool calls:
1. `query_record_info(user_id="...", record_type="MOODS")` — 注意不是query_expenses
Output: 展示心情记录列表
